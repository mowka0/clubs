# PRD: «Клубы» (Clubs) — Telegram Mini App

**Версия:** 1.1 (MVP — обновлён технический стек)
**Дата:** 05.03.2026
**Платформа:** Telegram Mini App
**Регион запуска:** Москва, Россия (СНГ)
**Язык интерфейса:** Русский

---

## 1. Обзор продукта и цели

### 1.1 Что мы создаём

«Клубы» — это Telegram Mini App, который позволяет создавать и вступать в платные офлайн-сообщества по интересам с системой репутации, двухэтапным подтверждением участия во встречах и мотивацией для организаторов.

### 1.2 Проблема

Современные приложения либо гиперцифровые (соцсети), либо транзакционные (агрегаторы событий). Люди испытывают «цифровое одиночество» и хотят принадлежности — не просто событий, а «своих людей» рядом. Telegram-чаты не решают эту проблему: в них нет структуры, ответственности и фильтрации случайных людей.

### 1.3 Решение

Платформа, где каждый клуб — это не просто чат, а социальный организм с правилами, репутацией и финансовой мотивацией. Гибридная архитектура: нативный Telegram-чат для общения + Mini App как «штаб-квартира» для управления.

### 1.4 Цели MVP

- Запустить рабочую платформу с полным циклом: создание клуба → вступление → организация встреч → отслеживание репутации.
- Проверить гипотезу: готовы ли люди платить за структурированные офлайн-сообщества.
- Собрать данные о поведении участников (надёжность, спонтанность) с первого дня.
- Привлечь первых 20–50 организаторов в Москве.

### 1.5 Метрики успеха

- Количество созданных клубов с ≥10 участниками.
- Retention участников (% оплативших второй месяц).
- Среднее количество встреч на клуб в месяц.
- Средний % посещаемости (подтвердил → пришёл).

## 2. Целевая аудитория

### 2.1 Участники

- Возраст: 20–40 лет.
- Локация: Москва (на старте).
- Профиль: Люди, которые хотят регулярно заниматься хобби в компании (футбол, настолки, кулинария, кино), но не могут стабильно собрать команду через обычные чаты.
- Мотивация: Найти «свою стаю» для регулярных офлайн-встреч.

### 2.2 Организаторы

- Активные лидеры существующих Telegram-чатов и сообществ.
- Профиль: Люди, которые уже организуют активности, но устали от хаоса в чатах (кто идёт, кто нет, кто подвёл).
- Мотивация: Монетизация организаторских усилий (80% от подписок), инструменты для управления, репутационная система для фильтрации «пустозвонов».

## 3. Архитектура системы

### 3.1 Гибридная модель

Система состоит из трёх компонентов, работающих как единое целое:

**Telegram-группа (супергруппа):** Нативный чат клуба. Участники общаются здесь в привычном формате. Группа создаётся автоматически при создании клуба или привязывается существующая.

**Telegram-бот:** Мост между чатом и Mini App. Живёт в группе, постит анонсы новых событий, отправляет напоминания, отвечает на команды (`/кто_идет`, `/мой_рейтинг`), вытягивая данные из бэкенда.

**Mini App (штаб-квартира):** Полноценный веб-интерфейс внутри Telegram. Содержит: каталог клубов (Discovery), календарь событий, профили с репутацией, панель организатора, управление подпиской.

### 3.2 Принцип взаимодействия

Пользователь почти не меняет привычек — он общается в обычном чате. Вся аналитика, репутация и управление живут в Mini App. Бот обеспечивает бесшовный переход: кнопки в сообщениях бота открывают нужные экраны Mini App.

## 4. Функциональные требования

### 4.1 Discovery (Каталог клубов)

#### 4.1.1 Главная лента

Лента представляет собой «афишу активностей города», а не скучный список.

**Карточка клуба содержит:**
- Название, категория, аватар/обложка.
- Город / район.
- Цена подписки (в месяц).
- Индикатор «Жизни»: количество участников, планирующих пойти на ближайшую встречу. Приоритет отображения — именно этот показатель, а не общее число подписчиков.
- Промо-теги: «Новый», «Популярный», «Свободные места», «Есть пробный период».
- Тип доступа (открытый / закрытый).

**Acceptance criteria:**
- Лента отображается при открытии Mini App.
- Карточки клубов сортируются по алгоритму (активность > новизна > рейтинг организатора).
- Индикатор «Жизни» обновляется в реальном времени на основе статусов участия в ближайшем событии.
- Промо-теги генерируются автоматически на основе данных (новый — создан < 2 недель назад; популярный — в топ-10% по соотношению заявок к местам; свободные места — заполнено < 80% лимита).

#### 4.1.2 Фильтрация и поиск

**Основные фильтры (видимы сразу):**
- Город (определяется автоматически по геопозиции или вводится вручную).
- Категория (Спорт, Творчество, Еда, Настольные игры, Кино, Образование, Путешествия, Другое).
- Тип доступа (Открытые / Закрытые).

**Расширенные фильтры (под кнопкой «Ещё»):**
- Цена подписки (диапазон, ползунок).
- Размер клуба (до 20 / 20–50 / 50+).
- День недели встреч (если организатор указал регулярность).

**Текстовый поиск:** По названию и описанию клуба.

**Acceptance criteria:**
- Геопозиция запрашивается при первом запуске; результат кешируется; пользователь может изменить город вручную.
- Фильтры применяются без перезагрузки страницы (клиентская фильтрация при малом количестве данных, серверная — при масштабировании).
- Комбинация фильтров работает по принципу AND.

#### 4.1.3 Страница клуба (детальная карточка)

При нажатии на карточку в ленте открывается экран с полной информацией:
- Название, обложка, описание (до 500 символов).
- Категория, город/район, тип доступа.
- Цена подписки.
- Количество участников / лимит.
- Ближайшее событие (название, дата, сколько идёт).
- Правила клуба (если заполнены).
- Кнопка действия: «Вступить» (открытый), «Хочу вступить» (закрытый), недоступен (приватный — не виден в каталоге).

### 4.2 Вступление в клуб (User Flows)

#### 4.2.1 Открытый клуб

1. Пользователь нажимает «Вступить».
2. Отображается экран подтверждения: цена, описание, правила (если есть, с галочкой «Ознакомился и согласен»).
3. Переход к оплате через Telegram Stars.
4. После успешной оплаты: пользователь автоматически добавляется в Telegram-группу клуба и получает доступ к календарю событий в Mini App.
5. Бот отправляет приветственное сообщение в группу.

**Acceptance criteria:**
- Весь flow занимает не более 3 шагов (кнопка → подтверждение → оплата).
- После оплаты пользователь автоматически добавляется в группу (через Telegram Bot API invite link).
- Запись в БД: membership со статусом `active`, дата вступления, дата следующего списания.

#### 4.2.2 Закрытый клуб

1. Пользователь нажимает «Хочу вступить».
2. Отображается форма с вопросом от организатора (например, «Расскажи о себе и почему хочешь к нам»).
3. Пользователь отправляет ответ.
4. Отображается статус: «Заявка отправлена. Обычно ответ в течение нескольких часов».
5. Организатор получает уведомление (пуш от бота) и видит заявку в панели управления.
6. Организатор нажимает «Принять» → запускается процесс оплаты для пользователя → после оплаты — добавление в группу.
7. Или организатор нажимает «Отклонить» (опционально с причиной) → пользователь получает уведомление.

**Автоматика антитоксичности:** Если заявка остаётся без ответа 48 часов, она автоматически отклоняется. Клуб получает понижение в рейтинге активности (влияет на позицию в ленте).

**Acceptance criteria:**
- Организатор получает push-уведомление через бота в личные сообщения в течение 1 минуты после подачи заявки.
- Статус заявки доступен пользователю в Mini App в любой момент.
- Cron-задача проверяет заявки старше 48 часов и отклоняет автоматически.
- При автоотклонении: рейтинг активности клуба уменьшается на фиксированное значение.

#### 4.2.3 Приватный клуб

1. Пользователь переходит по ссылке-приглашению (формат: `t.me/clubsapp?startapp=invite_XXXXX`).
2. Открывается Mini App с карточкой приватного клуба.
3. Процесс оплаты и вступления аналогичен открытому клубу (или с подтверждением, если организатор настроил).

**Acceptance criteria:**
- Приватные клубы не отображаются в каталоге ни при каких условиях.
- Инвайт-ссылка генерируется организатором в панели управления.
- Ссылка может быть одноразовой или многоразовой (настройка организатора).

### 4.3 Внутренний экран клуба (для участника)

После вступления участник видит «штаб-квартиру» клуба в Mini App:

#### 4.3.1 Календарь событий

- Список предстоящих и прошедших событий клуба.
- Для каждого события: название, дата/время, место (текст), лимит участников, текущий статус набора (сколько «Пойду» / «Возможно» / подтвердило).
- Кнопки выбора статуса: «Пойду» / «Возможно» / «Не пойду» (Этап 1 голосования).
- Отображение своего текущего статуса по каждому событию.

#### 4.3.2 Профиль участника

- Аватар (из Telegram), имя.
- Дата вступления в клуб.
- Три показателя репутации:
  - **Индекс надёжности** — общий числовой счётчик (сумма плюсов и минусов за всю историю).
  - **Процент выполнения обещаний** — (количество приходов после подтверждения / количество подтверждений) × 100%.
  - **Счётчик спонтанности** — количество раз, когда участник перешёл из «Возможно» в «Подтвердил» → «Пришёл».
- История посещений (список событий с отметками: был / не был).

#### 4.3.3 Список участников клуба

- Список всех участников с аватарами и индексом надёжности.
- Возможность открыть профиль любого участника.

### 4.4 Система «Живые Вклады» (Репутация и двухэтапное подтверждение)

Это ядро продукта — механика, которая превращает платформу из «платного календарика» в социальный организм.

#### 4.4.1 Этап 1: Предварительное голосование (за 3–5 дней до события)

**Триггер:** Организатор создаёт событие с указанием даты, лимита участников и параметра «за сколько дней открыть голосование» (по умолчанию — 5 дней).

**Действия:**
1. Бот отправляет уведомление в группу: «Новая активность: [Название]. Нужно [N] человек. Отметь свой статус».
2. Сообщение содержит кнопку, открывающую Mini App на экране события.
3. Участник выбирает статус:
   - ✅ **Пойду** — высокая мотивация, хочет место.
   - 🤔 **Возможно** — интересно, но не уверен; рассмотрит, если не наберётся основной состав.
   - ❌ **Не пойду** — не интересно / занят.

**Данные для организатора:**
- Количество в каждом статусе.
- Соотношение «Пойду» к лимиту (переизбыток / недобор).

**Acceptance criteria:**
- Уведомление в группу уходит автоматически при создании события.
- Участник может менять статус до начала Этапа 2.
- Статусы сохраняются в БД: таблица `event_responses` (user_id, event_id, stage_1_status, stage_1_timestamp).
- Mini App отображает текущую сводку (сколько «Пойду» / «Возможно» / «Не пойду»).

#### 4.4.2 Этап 2: Финальное подтверждение (за 24 часа до события)

Автоматически запускается cron-задачей за 24 часа до времени события.

**Сценарий А: Избыток (количество «Пойду» > лимита)**

1. Система отправляет уведомление ВСЕМ из статуса «Пойду»: «До [событие] остались сутки. Мест [N], желающих [M]. Подтверди участие прямо сейчас, чтобы занять место. Кто первый — тот в деле!»
2. Первые N человек, нажавших «Подтверждаю», попадают в финальный список.
3. Остальные автоматически попадают в лист ожидания (уведомление: «Вы в резерве»).
4. Если кто-то из финального списка отказывается, его место автоматически предлагается первому в листе ожидания.

**Сценарий Б: Недобор или равенство (количество «Пойду» ≤ лимита)**

1. Система запрашивает подтверждение у всех «Пойду».
2. После сбора подтверждений, если есть свободные места — система обращается к пулу «Возможно»: «Нужны ещё [X] человек. Если хочешь пойти — подтверждай. Кто первый — тот в деле!»
3. Из «Возможно» первые X подтвердивших попадают в финальный список.
4. Если после этого остаётся дефицит — организатор получает уведомление с вариантами: провести встречу в текущем составе, пригласить друзей извне (гостевой пропуск — будущая фича), отменить встречу.

**Acceptance criteria:**
- Cron-задача запускается ровно за 24 часа до события.
- Порядок подтверждения определяется по timestamp (FIFO — first in, first out).
- Переход из листа ожидания в финальный список происходит автоматически при отказе подтверждённого участника.
- Все переходы статусов логируются: таблица `event_responses` обновляется (stage_2_status, stage_2_timestamp, final_status).
- Уведомления рассылаются через бота в личные сообщения участникам.

#### 4.4.3 Отметка присутствия (после события)

**Триггер:** На следующий день после события организатору приходит push-уведомление: «Встреча [Название] прошла. Отметьте, кто пришёл».

**Процесс:**
1. Организатор открывает Mini App → экран прошедшего события.
2. Видит список финальных участников (кто подтвердил на Этапе 2).
3. Проставляет каждому: ✅ Пришёл / ❌ Не пришёл.
4. Нажимает «Сохранить».

**Защита от ошибок (оспаривание):**
1. Каждому участнику, отмеченному как «Не пришёл», отправляется уведомление: «Организатор отметил, что вы не были на [событие]. Если это ошибка — нажмите "Оспорить"».
2. Кнопка «Оспорить» открывает диалог с организатором в личных сообщениях Telegram.
3. Организатор может исправить отметку в течение 2 дней после первоначальной фиксации.
4. По истечении 2 дней результаты финализируются и влияют на репутацию.

**Acceptance criteria:**
- Push-уведомление организатору отправляется через 12 часов после времени окончания события.
- Уведомление об оспаривании отправляется участнику сразу после сохранения отметок.
- Окно оспаривания: 48 часов с момента сохранения отметок.
- После финализации результатов запускается пересчёт репутации.

#### 4.4.4 Начисление репутации

Репутация пересчитывается после финализации результатов каждого события (через 48 часов после отметки организатором).

**Таблица начисления:**

| Тип поведения | Этап 1 | Этап 2 | Приход | Надёжность | Спонтанность |
|---|---|---|---|---|---|
| Железобетонный | ✅ Пойду | ✅ Подтвердил | ✅ Пришёл | +100 | — |
| Пустозвон | ✅ Пойду | ✅ Подтвердил | ❌ Не пришёл | −50 | — |
| Передумавший | ✅ Пойду | ❌ Отказался | — | 0 | — |
| Спонтанный | 🤔 Возможно | ✅ Подтвердил | ✅ Пришёл | +30 | +1 |
| Вечный сомневающийся | 🤔 Возможно | ❌ Не подтвердил | — | 0 | — |
| Молчун | ❌ Не пойду | — | — | 0 | — |

**Формулы:**
- **Индекс надёжности** = Σ всех начислений за всю историю (может быть отрицательным).
- **Процент выполнения обещаний** = (количество «Пришёл» после «Подтвердил» / общее количество «Подтвердил») × 100%.
- **Счётчик спонтанности** = количество раз «Возможно» → «Подтвердил» → «Пришёл».

**Acceptance criteria:**
- Пересчёт запускается автоматически после закрытия окна оспаривания.
- Значения репутации хранятся как агрегированные поля в таблице `user_club_reputation` (пересчитываются, а не вычисляются на лету).
- Репутация считается отдельно для каждого клуба (участник может иметь разную репутацию в разных клубах).

### 4.5 Панель организатора

#### 4.5.1 Создание клуба

Пошаговая форма с полями:

| Поле | Тип | Обязательность | Ограничения |
|---|---|---|---|
| Название клуба | text | обязательно | до 60 символов |
| Город / Район | text + geo | обязательно | выбор из справочника или ручной ввод |
| Категория | select | обязательно | Спорт, Творчество, Еда, Настольные игры, Кино, Образование, Путешествия, Другое |
| Тип доступа | radio | обязательно | Открытый / Закрытый / Приватный |
| Лимит участников | number | обязательно | от 10 до 80 |
| Цена подписки | number | обязательно | минимум и формат зависят от валюты (Stars) |
| Описание | textarea | обязательно | до 500 символов |
| Аватар / Обложка | image upload | рекомендуется | формат: jpg/png, до 5 MB |
| Правила клуба | textarea | опционально | до 2000 символов |
| Вопрос для заявки (закрытый клуб) | text | опционально | до 200 символов |

**Калькулятор дохода:** При вводе цены и лимита отображается подсказка: «При [N] участниках вы будете зарабатывать [X] в месяц» (с учётом комиссии 20%).

**После создания:** Автоматически создаётся Telegram-супергруппа, бот добавляется в группу, организатор назначается администратором.

**Acceptance criteria:**
- Валидация всех обязательных полей перед отправкой.
- Калькулятор обновляется в реальном времени при изменении цены или лимита.
- Создание группы через Telegram Bot API происходит асинхронно (пользователь видит статус «Создаём ваш клуб...»).
- После создания организатор перенаправляется на экран своего клуба.

#### 4.5.2 Дашборд организатора

**Список участников:**
- Таблица: Аватар, Имя, Дата вступления, Статус подписки (активна / просрочена), Индекс надёжности, Процент выполнения обещаний.
- Кнопка «Написать» (открывает личный диалог в Telegram).

**Управление заявками (для закрытых клубов):**
- Входящие заявки: Имя, Ответ на вопрос, Дата подачи.
- Кнопки: «Принять» / «Отклонить» (с опциональным полем причины).
- Счётчик времени до автоотклонения (48 часов).

**Создание событий:**
- Форма: Название (text, обязательно), Дата и время (datetime, обязательно), Место (text, обязательно), Максимум участников (number, обязательно), Описание (textarea, опционально), За сколько дней открыть голосование (number, по умолчанию 5).
- После создания: событие появляется в календаре, бот постит анонс в группу.

**Дашборд события (перед встречей):**
- Статистика набора: сколько «Пойду», «Возможно», «Подтвердило» (после Этапа 2).
- Финальный список участников (с аватарками).
- Статус дефицита: «Собралось [N] из [M]», рекомендации при недоборе.

**Управление прошедшими событиями:**
- Список прошедших встреч.
- Экран отметки присутствия (список подтвердивших с переключателями «Пришёл» / «Не пришёл»).

**Финансы (базовый блок):**
- Количество активных участников.
- Оборот за текущий месяц (сумма подписок).
- Заработок организатора (80% от оборота).
- Дата ближайшего списания подписок.

### 4.6 Telegram-бот

#### 4.6.1 Функции бота в групповом чате

- Автоматическая публикация анонса при создании события (с кнопкой «Открыть в приложении»).
- Напоминание о голосовании (Этап 1) за N дней до события.
- Уведомление о запуске Этапа 2 (за 24 часа).
- Приветственное сообщение при добавлении нового участника.

#### 4.6.2 Команды бота

- `/кто_идет` — показывает текущий статус набора на ближайшее событие (сколько «Пойду», «Возможно», «Подтвердило»).
- `/мой_рейтинг` — показывает индекс надёжности, % выполнения и счётчик спонтанности текущего пользователя.
- `/события` — список предстоящих событий с кнопкой открытия Mini App.

#### 4.6.3 Личные уведомления (через бота в ЛС)

- Уведомление о новом событии в клубе.
- Напоминание о необходимости подтвердить участие (Этап 2).
- Уведомление о попадании в лист ожидания / освобождении места.
- Уведомление об отметке «Не пришёл» с кнопкой «Оспорить».
- Уведомление об одобрении / отклонении заявки (закрытые клубы).

**Acceptance criteria:**
- Бот отвечает на команды в течение 2 секунд.
- Все уведомления содержат inline-кнопку для перехода в Mini App.
- Бот корректно работает в нескольких группах одновременно (multitenancy).

### 4.7 Оплата и монетизация

#### 4.7.1 Модель

- Участник платит ежемесячную подписку за членство в клубе.
- 80% подписки получает организатор.
- 20% — комиссия платформы.

#### 4.7.2 Платёжная система

- Telegram Stars как единственный способ оплаты в MVP.
- Telegram Stars API поддерживает рекуррентные платежи (подписки).

#### 4.7.3 Жизненный цикл подписки

1. **Активация:** После оплаты первого месяца. Статус: `active`.
2. **Продление:** Автоматическое списание каждые 30 дней. Пользователь получает уведомление за 3 дня до списания.
3. **Неудачное списание:** Статус: `grace_period` (3 дня на пополнение баланса Stars). Пользователь получает уведомление.
4. **Отмена:** Пользователь может отменить подписку вручную. Доступ сохраняется до конца оплаченного периода. Статус: `cancelled`.
5. **Истечение:** Если подписка не продлена — пользователь теряет доступ к Mini App клуба, но остаётся в Telegram-группе (решение: убирать или нет — на усмотрение организатора). Статус: `expired`.

**Acceptance criteria:**
- Все транзакции логируются в таблице `transactions`.
- Организатор видит финансовую статистику в дашборде в реальном времени.
- При смене цены подписки действующие участники продолжают платить по старой цене до отмены.

**Принцип работы с данными (jOOQ):**

Проект НЕ использует ORM (Hibernate/JPA). Вместо этого применяется jOOQ — типобезопасный SQL-билдер. Рабочий цикл:

1. Разработчик пишет SQL-миграцию (Flyway): `V1__create_users.sql`.
2. Flyway применяет миграцию к БД.
3. jOOQ codegen plugin (Gradle) сканирует схему БД и генерирует Kotlin-классы: таблицы, поля, записи.
4. В репозиториях пишутся запросы через jOOQ DSL — они выглядят как SQL, но с типобезопасностью и автоподстановкой.
5. Результаты маппятся на Kotlin data-классы (DTO), а не на JPA-сущности.

Это обеспечивает полную прозрачность: каждый запрос — это видимый SQL, без lazy loading, кешей и скрытой магии.

## 7. Принципы UI-дизайна

### 7.1 Общие принципы

- **Нативность:** Интерфейс должен ощущаться как часть Telegram, а не как отдельное приложение. Использовать Telegram UI Kit, системные цвета и шрифты.
- **Эмоциональность:** Карточки клубов «продают эмоцию», а не показывают таблицу данных. Обложки, индикаторы жизни, теги — всё работает на вовлечение.
- **Минимализм действий:** Вступление в открытый клуб — максимум 3 нажатия. Подтверждение участия в событии — 1 нажатие.
- **Информативность без перегруза:** Показывать ключевые метрики (кто идёт, надёжность), но прятать детали в глубину (история посещений, полный профиль).

### 7.2 Ключевые экраны

1. **Discovery (главная лента):** Вертикальный скролл карточек, фильтры сверху, поиск.
2. **Карточка клуба (детальная):** Обложка сверху, описание, ближайшее событие, кнопка действия внизу (fixed).
3. **Внутренний экран клуба:** Табы — «События», «Участники», «Мой профиль».
4. **Экран события:** Статус набора, список участников, кнопка голосования / подтверждения.
5. **Профиль участника:** Три метрики репутации крупно, история посещений списком.
6. **Панель организатора:** Отдельный таб или раздел — управление клубом, события, участники, заявки, финансы.

## 8. Соображения по безопасности

### 8.1 Аутентификация

- Аутентификация через Telegram Mini App Init Data (проверка подписи `initData` от Telegram). Отдельной системы логин/пароль не требуется.
- Все API-запросы содержат JWT-токен, выданный после валидации `initData`.
- JWT включает: `user_id`, `telegram_id`, время выдачи, время истечения.

### 8.2 Авторизация

- Роли: `member`, `organizer` (per club).
- Организатор имеет доступ к панели управления только своих клубов.
- Участник видит только клубы, в которых состоит (внутренние экраны).
- API-эндпоинты защищены middleware проверки ролей.

### 8.3 Защита данных

- Хранение минимума персональных данных (только то, что даёт Telegram API).
- HTTPS для всех соединений.
- Параметризованные SQL-запросы (jOOQ генерирует prepared statements по умолчанию, SQL-инъекции исключены на уровне DSL).
- Rate limiting на API (через Spring фильтры или bucket4j, защита от брутфорса и спама заявок).

### 8.4 Защита от злоупотреблений

- Автоотклонение заявок через 48 часов (защита от игнорирования).
- Понижение рейтинга клуба при неактивности организатора.
- Лимит на количество клубов одного организатора (например, 10 в MVP).
- Лимит на количество заявок от одного пользователя в день.

- Команда бота: `/мой_рейтинг`.

### Milestone 5: Оплата и финансы (1–2 недели)
- Интеграция Telegram Stars API.
- Оплата подписки при вступлении.
- Рекуррентные платежи (автопродление).
- Жизненный цикл подписки (active → grace_period → expired).
- Финансовый блок в дашборде организатора.
- Распределение: 80% организатору, 20% платформе.

### Milestone 6: Панель организатора и polish (1–2 недели)
- Дашборд: список участников, управление заявками.
- Дашборд события: статистика набора, финальный список.
- Калькулятор дохода при создании клуба.
- Промо-теги в каталоге (автогенерация).
- Тестирование всех flow end-to-end.
- Исправление багов, оптимизация.

**Итого MVP: 9–15 недель.**

## 10. Потенциальные проблемы и способы их решения

| Проблема | Риск | Решение |
|---|---|---|
| Telegram Bot API ограничения на создание групп | Нельзя массово создавать группы через бот | Использовать привязку существующих групп как fallback. Организатор создаёт группу сам, бот привязывается через добавление. |
| Telegram Stars — ограниченная экосистема | Не все пользователи знакомы со Stars, возможны ограничения на вывод | Чёткий onboarding по оплате. В будущем добавить альтернативные методы. |
| Холодный старт: нет клубов → нет пользователей | Классическая проблема marketplace | Seed-стратегия: лично привлечь 20–50 организаторов в Москве. Создать шаблонные клубы. Партнёрства с существующими сообществами. |
| Организатор забывает отмечать присутствие | Данные репутации будут неполными | Push-напоминания. Повторное напоминание через 24 часа. В будущем — self-check-in участников через QR. |
| Токсичный организатор | Плохой опыт участников, отток | Механика «Исход» (будущее): автопонижение рейтинга при оттоке >30%. В MVP — возможность жалобы на клуб. |
| Масштабирование уведомлений | При росте нагрузки бот может упираться в rate limits Telegram | Redis очереди с rate limiting (Spring Data Redis). Batch-отправка. Webhook вместо polling. |

## 11. Возможности будущего расширения (за пределами MVP)

1. **Казна клуба:** Прозрачный бюджет с голосованием за траты (80% подписки → казна, организатор предлагает варианты).
2. **Амбассадоры входов:** Роль «Встречающего» для интеграции новичков (скидка на подписку или бонус к репутации).
3. **Цифровые сувениры:** Уникальные карточки-стикеры в профиль за каждую посещённую встречу (коллекционирование).
4. **Дикий конструктор клубов:** Пользователь предлагает активность, набирает людей с нуля (вирусный рост).
5. **Социальный рейтинг города:** Кросс-клубная репутация как «социальный паспорт» для привилегий у партнёров.
6. **Гостевой пропуск:** Приглашение друзей извне на конкретное событие (разовый доступ).
7. **Партнёрские интеграции:** Скидки при бронировании площадок, аренда инвентаря.
8. **Self-check-in:** QR-код на месте для автоматической отметки присутствия без участия организатора.
9. **Аналитика для организаторов (премиум):** Детальные данные об активности, retention, cohort analysis.
10. **Продвижение клубов (премиум):** Платное размещение в топе ленты Discovery.
